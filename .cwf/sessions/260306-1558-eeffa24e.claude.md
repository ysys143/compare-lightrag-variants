## EdgeQuake Chunk-Entity Storage Architecture

**Data Flow:**
1. **Document Ingestion** → Pipeline processes document, creating Chunks (MD5-hashed IDs like `chunk-{md5}`)
2. **Entity Extraction** → ExtractedEntity captures `source_chunk_ids: Vec<String>` linking entities back to chunks
3. **Graph Storage** → Entities become GraphNode instances with `source_chunk_ids` as node property
4. **Vector Storage** → Both entities and chunks stored separately with metadata type discrimination

**Key Insight: Chunks NOT Stored as Graph Nodes**

Unlike RAG-Anything which may store chunks as nodes, EdgeQuake:
- Stores chunks in **vector storage** with metadata `type: "chunk"` (orchestrator/ingestion.rs:290)
- Stores chunks in **KV storage** separately for content retrieval
- Stores chunks in **PostgreSQL as document records** (via pdf_storage_impl.rs)
- **Entities become graph nodes**, but **chunks are NOT nodes in the knowledge graph**
- Entities maintain `source_chunk_ids` as a **property array** linking to source chunks
